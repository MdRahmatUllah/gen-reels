# MinIO Storage Configuration

## Purpose

MinIO is the platform's default S3-compatible object storage backend. It stores all generated media assets, intermediate outputs, composed exports, quarantined content, and model weights (when self-hosted). This document defines the bucket structure, access policies, lifecycle rules, and operational configuration.

---

## Why MinIO

- **S3-compatible API:** All application code uses the standard S3 SDK (`boto3` or `minio-py`). Swapping MinIO for AWS S3, GCS, or any S3-compatible backend requires only endpoint and credential changes — no code changes.
- **Self-hosted:** Runs as a single Docker container for local development and small deployments. Scales to distributed mode for production.
- **Cost control:** No per-request egress charges in self-hosted mode. Significant cost savings for high-volume media generation workflows.
- **Local development parity:** Developers run the same storage interface locally as production, eliminating cloud-specific dev/prod drift.

## Bucket Structure

```text
reels-assets/
  workspace/{workspace_id}/project/{project_id}/
    briefs/
    scripts/
    scenes/
    consistency/
      pack.json
      reference_images/
      character_sheets/
    assets/
      images/
        start_frames/
        end_frames/
      videos/
        raw/
        silent/
      audio/
        narration/
        stripped/
      music/
      subtitles/
      uploads/
    exports/

reels-quarantine/
  {workspace_id}/{asset_id}/

reels-models/          (Phase 7: open-source model weights)
  wan2.1/
  flux/
  xttsv2/
  kokoro/

reels-temp/            (intermediate processing, auto-cleaned)
  {job_id}/
```

| Bucket | Purpose | Retention |
|---|---|---|
| `reels-assets` | All production media assets and exports | Plan-tier retention policy |
| `reels-quarantine` | Moderation-rejected content | 90 days minimum, then archival |
| `reels-models` | Open-source model weights (Phase 7) | Permanent (manual updates) |
| `reels-temp` | Intermediate FFmpeg processing files | Auto-deleted after 24 hours |

## Access Policies

### Application-Level Access

| Service | Bucket Access | Operations |
|---|---|---|
| API Service | `reels-assets` (read) | Generate pre-signed URLs for frontend |
| Workers | `reels-assets` (read/write) | Upload generated assets, read references |
| Workers | `reels-quarantine` (write) | Move rejected assets to quarantine |
| Workers | `reels-temp` (read/write) | Store/retrieve intermediate processing files |
| Composition Worker | `reels-assets` (read), `reels-temp` (read/write) | Read scene assets, write intermediates, write final export |

### Pre-Signed URL Policy

All client-facing asset access uses pre-signed URLs generated by the API service:

| Use Case | URL TTL | Method |
|---|---|---|
| In-app asset preview | 1 hour | GET |
| Export download | 5 minutes | GET |
| Worker upload | Step deadline duration | PUT |
| Reference image for generation | 30 minutes | GET |
| User media upload | 15 minutes | PUT |

**Rule:** No public bucket policies. All buckets are private. All client access is through pre-signed URLs.

## MinIO Configuration

### Docker Compose (Local Development)

```yaml
minio:
  image: minio/minio:latest
  ports:
    - "9000:9000"   # S3 API
    - "9001:9001"   # Web Console
  command: server /data --console-address ":9001"
  environment:
    MINIO_ROOT_USER: minioadmin
    MINIO_ROOT_PASSWORD: minioadmin
  volumes:
    - minio_data:/data
  healthcheck:
    test: ["CMD", "mc", "ready", "local"]
    interval: 10s
    timeout: 5s
    retries: 5
```

### Bucket Initialization Script

Run once after first `docker compose up`:

```bash
#!/bin/bash
# scripts/init-minio.sh

# Wait for MinIO to be ready
until mc alias set local http://minio:9000 minioadmin minioadmin; do
  echo "Waiting for MinIO..."
  sleep 2
done

# Create buckets
mc mb local/reels-assets --ignore-existing
mc mb local/reels-quarantine --ignore-existing
mc mb local/reels-temp --ignore-existing

# Set lifecycle policy on temp bucket (auto-delete after 1 day)
mc ilm rule add local/reels-temp --expire-days 1

echo "MinIO buckets initialized."
```

### Application Configuration

```python
# core/config.py
class StorageSettings:
    MINIO_ENDPOINT: str = "minio:9000"
    MINIO_ACCESS_KEY: str = "minioadmin"
    MINIO_SECRET_KEY: str = "minioadmin"
    MINIO_SECURE: bool = False  # True for production with TLS
    MINIO_REGION: str = "us-east-1"

    BUCKET_ASSETS: str = "reels-assets"
    BUCKET_QUARANTINE: str = "reels-quarantine"
    BUCKET_TEMP: str = "reels-temp"
    BUCKET_MODELS: str = "reels-models"

    PRESIGN_TTL_PREVIEW: int = 3600       # 1 hour
    PRESIGN_TTL_DOWNLOAD: int = 300       # 5 minutes
    PRESIGN_TTL_UPLOAD: int = 900         # 15 minutes
    PRESIGN_TTL_REFERENCE: int = 1800     # 30 minutes
```

### SDK Usage

```python
from minio import Minio

client = Minio(
    endpoint=settings.MINIO_ENDPOINT,
    access_key=settings.MINIO_ACCESS_KEY,
    secret_key=settings.MINIO_SECRET_KEY,
    secure=settings.MINIO_SECURE,
    region=settings.MINIO_REGION,
)

# Upload an asset
client.put_object(
    bucket_name="reels-assets",
    object_name=f"workspace/{workspace_id}/project/{project_id}/assets/images/start_frames/{asset_id}.png",
    data=image_bytes,
    length=len(image_bytes),
    content_type="image/png",
)

# Generate pre-signed download URL
url = client.presigned_get_object(
    bucket_name="reels-assets",
    object_name=object_path,
    expires=timedelta(hours=1),
)
```

## Lifecycle And Retention Policies

| Asset Type | Retention | Auto-Action |
|---|---|---|
| Final exports | Plan-tier defined (default: 90 days) | Notify before deletion |
| Approved frame pairs | Same as exports | Retained with exports |
| Intermediate assets (raw video, stripped audio) | 30 days after export completion | Auto-delete |
| Temp files (`reels-temp`) | 24 hours | Auto-delete via lifecycle rule |
| Quarantined content | 90 days minimum | Archive to cold storage |
| Model weights (`reels-models`) | Permanent | Manual management |

## Production Considerations

### Scaling MinIO

- **Single-node:** Suitable for development and small deployments (< 1 TB).
- **Distributed mode:** 4+ nodes with erasure coding for production redundancy.
- **Cloud alternative:** Replace MinIO with AWS S3, GCS, or Azure Blob Storage by changing endpoint and credentials only.

### Backup Strategy

- MinIO supports `mc mirror` for bucket-level replication.
- For production, enable versioning on `reels-assets` and `reels-quarantine` buckets.
- Database records always contain the canonical asset reference — object storage is the data store, PostgreSQL is the index.

### TLS

- Local development runs MinIO without TLS (`MINIO_SECURE=False`).
- Production MinIO instances must use TLS. Set `MINIO_SECURE=True` and configure certificates via environment variables or a reverse proxy.
